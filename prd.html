PRD: HADR earthquake monitor

The self-correcting 08:30 sitrep

Scope: earthquakes, end to end (USGS · GDACS · ReliefWeb) Stage: build-plan-product · step B Status: ready-for-agent Issue: #3 Basis: ADR-0001–0008 · CONTEXT.md

Step B of build-plan-product. The architecture is already fixed by ADR-0001–0008 and CONTEXT.md; this PRD resolves the five product/policy questions that CONTEXT.md deferred "for the PRD" and specifies the product behaviour. Scope for this build: earthquakes, end to end across the three feeds. Other GDACS hazard types are out of scope for the build; the design is meant to generalise to them later.

Two runtime decisions taken during review are folded in: the scheduled ticks run hybrid (08:30 brief on GitHub Actions, fast tick as a harness agent), and urgent alerts are confirmation-only (no magnitude-only push).

Problem Statement

I coordinate disaster response, and in the first hours of an earthquake the thing that fails me is not effort — it is information. What I need in order to act (what happened, where, how bad, who is affected) is scattered across several authoritative feeds, arrives faster than I can read it, and is thick with duplicates and false alarms: the same quake reaches me three times under three different identifiers, sitting in a list alongside dozens of minor tremors that look exactly like it.

Worse, these feeds are not bulletins — they are mutable databases. A magnitude is revised down a day later; an alert colour is downgraded; an event is deleted outright. Every tool I have reports once and never looks back, so I am never told that something I acted on yesterday is no longer true. And none of this waits for office hours: the events I most need to know about happen overnight, in other time zones, when no one is on watch.

I cannot keep a human staring at three feeds around the clock — and even if I could, they could not reliably de-duplicate, filter, and (hardest of all) track what has quietly changed underneath a report they already trusted.

Solution

An unattended monitor that watches USGS, GDACS and ReliefWeb continuously and keeps one reconciled picture of active earthquake events — so the same quake arriving from three feeds is one canonical event, and minor noise is filtered out before it reaches me.

It behaves in three ways (ADR-0002):

  • Silent by default. It keeps its picture current without interrupting me. "Noticed within minutes" means recorded in the ledger (on the next tick), not alerted.
  • Urgent when it must be. A severe, high-confidence event breaks the silence immediately with a push notification, before the morning brief (ADR-0007).
  • A briefing every morning at 08:30 Singapore time. One decision-ready situation report, published to dashboard.html.

Its distinctive promise: the briefing leads with what changed since the last brief — including what it previously told me that is no longer true (new, upgraded, downgraded, corrected, retracted). And it states plainly what it cannot see, so I never mistake its silence for the absence of a disaster.

User Stories

Detection & coverage
  1. As a coordinator, I want authoritative earthquake feeds watched around the clock, so that a significant quake is noticed within minutes whether or not anyone is on duty.
  2. As a duty officer, I want the monitor to run unattended on a schedule, so that overnight and cross-time-zone events are not missed because a human was asleep.
  3. As a coordinator, I want a quake detected by USGS within minutes to already be in the picture by the time GDACS and ReliefWeb catch up, so that I benefit from the fastest feed without waiting for the slowest.
  4. As an analyst, I want each morning brief to state plainly what the monitor cannot see (floods, landslides, heat, conflict; that global completeness is only ~M4.5–5.0; that tsunami:1 is a flag, not a NOAA tsunami warning), so that I never mistake its silence for safety.
  5. As a coordinator, I want the brief to say when a feed was unreachable at run time, so that a gap in the data is visible rather than hidden.
Cutting the noise & de-duplicating
  1. As a coordinator, I want the same physical earthquake arriving from USGS and GDACS to appear as one canonical event, not two, so that my picture is not double-counted.
  2. As an analyst, I want a quake whose USGS identifier set or preferred id changes over its life to remain the same canonical event, so that revisions don't spawn phantom duplicates.
  3. As a coordinator, I want routine minor tremors (e.g. Green ~M4.5 quakes with no impact signal) filtered out of the headline, so that my attention is spent only on events that could matter.
  4. As an analyst, I want a ReliefWeb disaster record tied back to the GDACS/USGS event it describes (via GLIDE when present, or a recorded fuzzy match otherwise), so that humanitarian context attaches to the right event.
  5. As a coordinator, I want de-duplication and filtering to be reproducible, so that the same inputs always yield the same picture and I can trust it.
Provisional vs. confirmed
  1. As a duty officer, I want a brand-new quake shown as provisional until an impact signal lands, so that I understand an early read (magnitude/location only) may still move.
  2. As a coordinator, I want an event promoted to confirmed once PAGER / GDACS impact scoring (or a reviewed USGS status) arrives, so that I know when a read has firmed up.
  3. As an analyst, I want the brief to never present a provisional read as if it were settled, so that I don't over-commit resources on a number that has not been reviewed.
Urgent alerts (breaking silence)
  1. As a duty officer, I want an immediate push when a severe, high-confidence earthquake appears (GDACS Red, or USGS PAGER Orange/Red), so that I can act before 08:30.
  2. As a duty officer, I want not to be paged for a strong-but-unconfirmed quake with no impact signal yet, so that I am not woken for a deep offshore M7 that affects no one.
  3. As a duty officer, I want at most one push per event unless its severity genuinely escalates (e.g. Orange→Red), so that a stream of episode updates does not become alert spam.
  4. As a duty officer, I want downgrades and retractions to never arrive as an urgent push, so that only genuinely urgent, actionable news breaks my silence (corrections wait for the 08:30 brief).
  5. As a coordinator, I want the urgent-alert decision to be made deterministically (never dependent on the LLM), so that whether I get paged is reproducible and defensible.
The 08:30 briefing
  1. As a coordinator, I want one situation report published to dashboard.html at 08:30 Singapore time, so that I start the day with a shared, current picture.
  2. As a coordinator, I want the brief to open with what changed since the last brief — new / upgraded / downgraded / corrected / retracted — so that I read the delta first, not re-read yesterday.
  3. As a coordinator, I want each active event characterised by impact (expected affected population, alert colour, PAGER loss bins, humanitarian signals), not magnitude alone, so that I prioritise by consequence, not by number.
  4. As a coordinator, I want ongoing-but-unchanged events summarised below the fold rather than re-headlined, so that the top of the brief is only what's new or moved.
  5. As a coordinator, I want a clear "no material change" brief on a quiet morning (and no push), so that quiet is stated deliberately rather than left ambiguous.
  6. As an analyst, I want each event to link out to its sources (USGS/GDACS/ReliefWeb) with proper attribution, so that I can drill into the primary record.
Corrections & retractions (the distinctive feature)
  1. As a coordinator, I want a correction stated explicitly as "we said X → it is now Y → because the source revised it," so that I can trust the monitor precisely because it admits what changed.
  2. As a coordinator, I want a magnitude revision (e.g. M6.8 → M6.1) surfaced as a correction with its impact re-assessed, so that I can scale back a response that no longer fits.
  3. As a coordinator, I want an event the source positively withdrew (a USGS deletion, or a GDACS record removed) reported as a retraction, so that I stop acting on something no longer real — while a mere colour drop (even to Green) is reported as a downgrade, not a retraction.
  4. As an analyst, I want an event that merely aged out of a feed's window reported as "no longer in feed window; not confirmed ended" — distinct from a retraction — so that I don't read normal windowing as an all-clear.
  5. As a coordinator, I want corrections attributed to the feed's revision (the mutability model), not phrased as the monitor's mistake, so that the wording builds trust rather than doubt.
Operating unattended & trustworthy
  1. As a maintainer, I want the monitor to poll politely within each feed's limits (ReliefWeb ≤1000 calls/day; USGS gzip + If-Modified-Since; GDACS delta-poll on datemodified), so that it doesn't get rate-limited or blocked.
  2. As a maintainer, I want a feed outage (503, invalid XML, timeout) to degrade gracefully — keep the last known state for that feed and note the gap — rather than crash or corrupt the ledger.
  3. As a maintainer, I want state to persist durably between scheduled runs, so that "what changed since yesterday" survives across stateless cloud invocations.
  4. As a reviewer, I want a human-readable record of what each briefing asserted, so that I can audit past briefs even though the ledger DB itself diffs opaquely in git.
  5. As a maintainer, I want the expensive model step to run only when the deterministic check says something changed, so that a scheduled run that does nothing costs neither money nor trust.
  6. As a maintainer, I want ReliefWeb report bodies handled as third-party copyright (link + short excerpt + attribution only), so that the monitor respects licensing and the AI-content clause.
  7. As a reviewer, I want every LLM-made fuzzy link decision recorded in the ledger and overridable, so that a mis-link can be corrected and the picture stays reproducible thereafter.
  8. As a maintainer, I want the deterministic core to be exercisable from recorded feed fixtures with a frozen clock, so that I can prove behaviour without hitting the network or waiting for real events.
  9. As an operator, I want to run the whole pipeline on demand (not only on the schedule), so that I can validate a change end to end before trusting it overnight.
  10. As a coordinator, I want the brief to carry a clear "as of" timestamp in Singapore time, so that I know exactly how current the picture is.
  11. As a maintainer, I want the schedule to be a lightweight ~30-minute poll+triage tick plus the 08:30 briefing run, so that operating cost matches the two very different cost profiles of ingest vs. briefing.

Implementation Decisions

Runtime split (ADR-0001, ADR-0008)

  • The deterministic core (Python) owns the hot path end to end: fetch, parse, cross-feed identity, the ledger, change detection, the noise filter, and the urgent-alert decision and message. Same input → same ledger state.
  • The LLM/agent layer is confined to judgement + prose, and only at 08:30: impact assessment, the fuzzy cross-feed tie-break, and writing the brief. No LLM on the urgent path.
  • Modules by responsibility (per ADR-0008): a fetcher (feeds → normalised records), a matcher (records → canonical_id), a ledger (reconcile + change-detect + persist), and a briefer (diff + render). A thin change-detection/triage step decides whether the model runs at all.

Scope & the three layers

USGS = detection (earthquakes, minutes), GDACS = triage (impact-scored colour), ReliefWeb = confirmation/context (curated prose). The build wires all three, for earthquakes only.

Cross-feed identity — canonical_id (ADR-0004)

Two tiers. Deterministic (all in code, covers the fast path): union of USGS ids; GDACS (eventtype, eventid); the GDACS source/sourceid → USGS id crosswalk (NEIC-sourced quakes are near-exact); GLIDE → ReliefWeb↔GDACS; and spatiotemporal blocking (same hazard, time window, distance, magnitude tolerance) with thresholds in config. LLM tie-break at 08:30 only: the fuzzy residual (mostly ReliefWeb prose ↔ hazard events) is resolved by the model, its decision written to the ledger and overridable. The urgent-alert path is fully deterministic and never varies with the LLM.

Ledger model & write coordination (ADR-0003, ADR-0006)

One row per canonical event holding the latest reconciled values, plus a stored published snapshot of what each briefing asserted. "What changed since yesterday" = diff current state vs. the last published snapshot. Persisted as SQLite committed to the repo (state/ledger.db). Because the DB diffs opaquely in git, each briefing is also committed as a readable file (state/published/<date>.json and/or rendered markdown) to preserve an inspectable audit trail.

Write coordination (refines ADR-0006). ADR-0006 assumed a single schedule serialised all writes; under the hybrid runtime (below) two schedulers run, so that assumption no longer holds — and a binary .db cannot be git-merged. Therefore the fast tick is the sole writer of ledger.db and commits it only when the reconciled state actually changed; the 08:30 brief only reads the ledger and writes disjoint paths (state/published/<date>.json, dashboard.html), and the LLM's fuzzy-link decisions are emitted to a disjoint file the next tick applies. Every commit does pull---rebase-and-retry; touching disjoint paths keeps the rebase conflict-free.

Canonical-event record (shape, not code): assigned canonical_id; hazard type; first-seen / last-updated; status ∈ {provisional, confirmed, retracted, aged_out}; severity fields — GDACS alertlevel (event-max, i.e. "was it ever severe") and episodealertlevel (latest, authoritative for current severity), PAGER alert, magnitude, depth; the last alert level pushed (so the one-push-per-event rule survives stateless ticks); geometry + origin time; per-feed references (USGS ids set, GDACS eventid+episodeid, ReliefWeb id + GLIDE); recorded LLM link decisions; and per-feed provenance / last-modified.

Q1 Hazard scope — earthquakes, end to end

USGS + GDACS + ReliefWeb, quakes only, at full depth (dedup, provisional→confirmed, corrections/retractions, fuzzy ReliefWeb linking). Chosen because the cross-feed identity story is cleanest for quakes (the GDACS source/sourceid crosswalk) and the fast alert path is almost entirely USGS↔GDACS quakes. Other GDACS hazards are out of scope for the build.

Q2 Urgent-alert threshold — "severe + high-confidence"

  • Severe is impact-based, not magnitude-based: GDACS latest-episode alert Red (episodealertlevel, not the monotonic event-max alertlevel), or USGS PAGER alert Orange or Red.
  • High-confidence = the impact signal has settled and the event is confirmed (not provisional). For PAGER, Orange/Red are human-reviewed before release, so their presence is the confidence. For GDACS Red, the parallel bar is a settled ShakeMap / episode impact score (GDACS EQ alerts wait up to ~25 min for ShakeMap) — an early, pre-ShakeMap Red is still provisional and does not fire. Raw magnitude alone never fires.
  • No extreme-magnitude escape hatch (per decision): a strong-but-unconfirmed quake does not push. Accepted tradeoff — up to ~20–40 min delay on the rare catastrophe in exchange for near-zero false alarms.
  • De-duplicated & escalation-only: at most one push per canonical event (enforced via the persisted last alert level pushed, so it holds across stateless ticks); push again only on a genuine escalation (e.g. Orange→Red). Downgrades and retractions never push — they wait for the 08:30 brief.
  • Both the fire/no-fire decision and the push message are deterministic (ledger facts in a fixed template); no LLM on the urgent path (ADR-0001, ADR-0004, ADR-0007). Narrative impact assessment is a briefing-time activity (Q4).

Q3 Noise filter & provisional→confirmed promotion

  • Enters the ledger (becomes a canonical event) when it clears a materiality gate using each feed's own layer semantics: present in GDACS (any colour) or ReliefWeb (curated ⇒ it matters), or from USGS at magnitude ≥ ~M4.5 (threshold in config). The ~M4.5 floor is chosen because USGS global completeness is only ~M4.5–5.0 — below it, absence is unreliable, so a lower floor would persist noise without buying coverage. Sub-threshold USGS micro-quakes are dropped, not persisted.
  • Green/low events that clear the gate are still persisted (just not featured) — precisely so a later escalation (Green→Orange→Red) is detectable as a change against the last snapshot. Persist-to-detect, headline-by-impact.
  • Provisional on first detection (USGS status=automatic; GDACS before ShakeMap/impact scoring; PAGER=null). Confirmed when an impact signal settles: PAGER non-null, GDACS episode carries settled impact scoring/ShakeMap, or USGS status flips automatic→reviewed. (Expect provisional for the first tick or two: PAGER is null ~20–40 min; GDACS EQ alerts wait up to ~25 min for ShakeMap.)
  • Alert gating uses confirmed severity only; the headline of the brief is drawn from material events, with routine Green quakes summarised, not featured.

Q4 Impact assessment & enrichment (briefing-time)

At briefing time (08:30 only — the urgent push carries only deterministic facts, Q2), the LLM layer enriches each surviving canonical event by pulling: USGS PAGER (fatality/economic-loss probability bins, population exposure), ShakeMap (shaking intensity / MMI and population within intensity bands), GDACS event detail (url.details, impact scoring, affected country/population), and the ReliefWeb narrative (short excerpt + link) for humanitarian context. Severity is characterised in impact terms and the brief states the basis (e.g. "GDACS Red; PAGER Orange — est. N exposed at MMI ≥ VI"). The numbers cited come from the pulled products; the deterministic ledger owns which events are in scope and their reconciled identity/severity.

Q5 Change classification & retraction wording

The briefing opens with a "Since the last brief" section computed as the diff of current ledger state vs. the last published snapshot, classified by impact tier (GDACS episodealertlevel + PAGER):

  • New canonical event present now, absent from the last snapshot.
  • Upgraded present in both; impact tier increased (episode colour up, PAGER up, or provisional→confirmed-severe).
  • Downgraded present in both; impact tier decreased. A colour drop — including all the way to Green — is a Downgrade, not a retraction (the event is real, just reassessed as minor).
  • Retracted was in the last snapshot; positively withdrawn by the source — a USGS deletion, or a GDACS record removed. A pure magnitude revision that doesn't move the tier is surfaced as a correction on the event, not a re-ranking.
  • Aged out no longer in a feed's window but not confirmed ended (a ~4-day GDACS drop-out ≠ event over). On the ReliefWeb RSS feed we cannot tell a withdrawal from a window scroll-out or a normal alert→current/past transition, so a disappeared ReliefWeb item is treated as aged-out unless corroborated — never a confirmed retraction (that needs the API status field, deferred).
  • Ongoing present in both, unchanged — folded below the fold, not re-headlined.

If the diff is empty, the 08:30 brief still publishes deterministically as "no material change since the last brief" with the LLM step gated off — quiet is stated, not silent-by-absence.

Corrections are phrased for trust: "we said X → it is now Y → because the source revised it," attributed to the feed's revision (the mutability model), never as the monitor's own error where the feed changed underneath it. A previously reported event is never silently dropped.

Scheduling & delivery (hybrid; refines ADR-0005, ADR-0007)

Two schedulers, split by what each needs:

  • The 08:30 Asia/Singapore briefing runs in GitHub Actions (the scaffolded sitrep.yml) — it publishes dashboard.html, needs no push, and expresses the deterministic gate cleanly. (08:30 SGT = 30 0 * * * UTC; Singapore has no DST.)
  • The ~30-minute poll+triage tick runs as a harness scheduled cloud agent, because the urgent push (ADR-0007) needs a Claude Code agent context that CI does not provide. It shells out to the deterministic Python and does minimal agent reasoning. This hybrid refines ADR-0005, which assumed one mechanism.

The gate is load-bearing: a deterministic change-check decides whether anything changed; the headless model runs only if it did — the model never decides whether to wake up. Detection latency is bounded by the tick interval, so "noticed within minutes" means recorded promptly on the next ~30-min tick, not sub-minute.

Delivery. Urgent alerts go out via the harness's own push notification, to the operator only (who, in this build, is also the coordinator; team broadcast is deferred). The push is best-effort — no delivery ack or retry.

Secrets. The Actions brief needs a model API credential as a repo secret plus write access to commit (GITHUB_TOKEN covers the commit); the harness tick's secrets live in its agent environment (ADR-0005). This is the one place the "no-secrets" framing of ADR-0005/0007 actually needs a secret.

Testing Decisions

What makes a good test here: it asserts external behaviour at a high seam, not implementation detail; it is deterministic given recorded inputs and a frozen clock; and it makes no assertion on generated prose. Two seams, because the deterministic core (assert exact state) and the LLM layer (inherently non-deterministic) need different regimes.

Seam 1 (primary): the feed-fetch boundary

Everything downstream of parsed feed records is pure. Tests inject recorded feed fixtures (GDACS EVENTS4APP JSON, USGS all_day.geojson, ReliefWeb RSS) plus a frozen clock, run the whole deterministic pipeline, and assert on: reconciled ledger state (canonical events, statuses, severity), the change-classification (new/upgraded/downgraded/retracted/aged-out), the noise filter, provisional→confirmed transitions, and the urgent-alert decision + message as data (a record/return value — no real push, no network). This is the highest deterministic seam and covers the hot path end to end. The push channel and the git commit are dumb sinks, verified as the upstream decision, not exercised for real.

Fixture-driven scenarios (each an external-behaviour test), driven as multi-tick sequences where mutation matters:

  1. Same quake from USGS + GDACS (GDACS source=NEIC, sourceid=USGS id) ⇒ one canonical event.
  2. USGS magnitude revision M6.8 → M6.1 across ticks ⇒ ledger updates; surfaced as a correction (tier unchanged) or Downgraded (tier drops).
  3. USGS ids set / preferred id changes ⇒ still one canonical event (union of ids).
  4. USGS deletion ⇒ Retracted; a GDACS record removed ⇒ Retracted (positive withdrawal only).
  5. GDACS episodealertlevel Orange → Red ⇒ Upgraded + urgent-alert fires once, and does not re-fire on the next tick (the pushed level is persisted).
  6. GDACS episode Red → Green ⇒ Downgraded; no push; not a retraction; surfaces only at 08:30.
  7. Green ~M4.5 swarm ⇒ persisted but filtered from the headline; no alert.
  8. Provisional strong quake (PAGER=null, or a pre-ShakeMap GDACS Red) ⇒ provisional, no alert; when PAGER Orange / a settled ShakeMap lands next tick ⇒ confirmed + alert.
  9. Event leaves the GDACS ~4-day window ⇒ aged out ("not confirmed ended"); a ReliefWeb item that disappears from RSS is likewise aged-out, not retracted.
  10. Feed returns 503 / invalid XML ⇒ pipeline degrades; that feed's state unchanged; the gap is recorded for the brief.
  11. Nothing changed since the last snapshot ⇒ the brief publishes a deterministic "no material change" state with the model gated off; no push.
  12. ReliefWeb GLIDE ties to a GDACS/USGS event ⇒ deterministic GLIDE join; a prose-only ReliefWeb item ⇒ given a recorded LLM link decision, the merge reconciles correctly.

Seam 2: the LLM boundary at 08:30

The briefing run computes a deterministic diff (tested at Seam 1) and hands it to a stubbed model. Tests assert the brief-assembly wiring (diff → the inputs handed to the model → the structure of the rendered dashboard.html: the "Since the last brief" buckets, the "cannot see" disclosure, source links) and that a written fuzzy-link decision reconciles as in scenario 12 — never the prose itself.

Modules under test & prior art

Under test: the deterministic pipeline from the fetch boundary inward, plus the brief-assembly wiring with a stubbed model. Prior art: none in-repo — this is greenfield, so this PRD establishes the fixture-driven, frozen-clock, golden-ledger pattern as the house style. It aligns with the existing house rule in scripts/README.md ("anything that must give the same answer twice does not belong in a prompt"). Learnings that cost >10 minutes are captured in docs/solutions/ per that directory's convention.

Out of Scope

  • Non-earthquake hazards (cyclones, floods, volcanoes, drought, wildfire). The architecture is meant to generalise, but they are not built this week.
  • Tsunami warnings (those are NOAA / tsunami.gov; tsunami:1 is a flag, not a warning), landslides, heat, conflict — not within the three feeds' EQ scope.
  • A magnitude-only urgent escape hatch — extreme quakes still wait for a confirmed impact signal before pushing (confirmation-only, by decision).
  • ReliefWeb v2 API integration (rich querying/filters, and the status field needed to confirm a ReliefWeb retraction) until an appname is approved. The build runs on the RSS feed (no approval needed); the API is a drop-in behind the same normalised record once approved.
  • Wholesale republishing of ReliefWeb report bodies (third-party copyright) — link + short excerpt + attribution only.
  • Team / multi-recipient alerting (email, Slack/Teams). The urgent alert reaches the operator only via harness push (ADR-0007); broadcast is a noted migration path.
  • External managed state store and a full event-sourced / append-only audit log (ADR-0003, ADR-0006 deliberately defer both).
  • Historical backfill beyond the feeds' live windows.
  • Auth / hosting for dashboard.html — it is a committed static artifact.
  • An always-on host + system cron (ADR-0005 defers; natural migration target).
